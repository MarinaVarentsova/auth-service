- существующую логику авторизации администратора.

artifacts/api-server

app/api/auth/register/route.ts

Фактическая интеграция UniSender находится в:

lib/unisender.ts

Текущая проблема:

1. POST /api/auth/register создаёт пользователя в auth_users.
2. Создаётся пользователь со статусом ожидания подтверждения.
3. Письмо не приходит.
4. API возвращает 502:
   "Пользователь создан, но письмо подтверждения не отправлено".
5. Созданный пользователь остаётся в БД.

Нужно провести аудит реального кода текущего репозитория и исправить проблему.

ШАГ 4. В lib/unisender.ts поддержи оба возможных успешных ответа UniSender.

Вариант 1:

{
  "result": [
    {
      "id": "message-id",
      "email": "user@example.com"
    }
  ]
}

Вариант 2:

{
  "result": {
    "email_id": 14362456134
  }
}

email_id может быть string или number.

Не считать успешный object-ответ ошибкой.

ШАГ 5. Добавь безопасное серверное логирование:

[AUTH-REGISTER] BUILD_MARKER_AUTH_SERVICE_V3
[AUTH-REGISTER] user created
[AUTH-REGISTER] verification token created
[EMAIL] verification request started
[EMAIL] verification response
[EMAIL] verification accepted
[EMAIL] verification failed
[AUTH-REGISTER] rollback completed
[AUTH-REGISTER] rollback failed

В ответе UniSender логировать:

- HTTP status;
- data.error;
- data.code;
- resultFormat: array, object или unknown;
- наличие message id;
- замаскированный email.

Никогда не логировать:

- UNISENDER_API_KEY;
- пароль;
- полный verification token;
- полный verificationUrl с токеном.

ШАГ 6. Исправь rollback.

Если пользователь и verification token были созданы в рамках текущего POST-запроса, но sendVerificationEmail завершился ошибкой:

1. удалить созданный verification token;
2. удалить только созданного в этом запросе пользователя;
3. не удалять ранее существующие записи;
4. не оставлять email заблокированным для повторной регистрации.

Не держать транзакцию базы открытой во время HTTP-запроса к UniSender.

При ошибке отправки вернуть:

HTTP 502

{
  "message": "Не удалось отправить письмо подтверждения. Попробуйте зарегистрироваться ещё раз.",
  "buildMarker": "auth-service-v3"
}

Поле buildMarker временное, нужно для проверки, что production действительно запустил новую версию.
