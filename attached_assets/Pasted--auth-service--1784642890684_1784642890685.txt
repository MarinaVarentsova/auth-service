Работаем только с сервисом регистрации и авторизации auth-service.

Не трогать основной сервис личного кабинета без необходимости.

Нужно закрыть две задачи.

==================================================
ЗАДАЧА 1. UX восстановления пароля
==================================================

Серверная логика восстановления пароля уже работает:

- письмо приходит;
- ссылка открывается;
- новый пароль сохраняется;
- токен проверяется корректно.

Проблема только в интерфейсе.

Сейчас при ошибке пользователь видит:

HTTP 400

Нужно исправить страницу reset-password.

Требования:

1. При запросе к /api/auth/reset-password всегда читать JSON ответа.

2. Если API вернул:

{
  "success": false,
  "error": "текст ошибки"
}

показывать пользователю значение поля error.

3. Не показывать пользователю технические сообщения:

- HTTP 400;
- HTTP 500;
- Bad Request;
- Internal Server Error.

4. Для недействительного, использованного или просроченного токена показывать понятный текст:

«Ссылка восстановления недействительна или уже была использована. Откройте самое последнее письмо или запросите восстановление пароля повторно.»

5. Для неизвестной ошибки показывать:

«Произошла ошибка. Попробуйте ещё раз.»

6. Во время отправки запроса:

- заблокировать кнопку;
- исключить повторное нажатие;
- показывать состояние загрузки.

7. После успешного изменения пароля:

- показать сообщение «Пароль успешно изменён»;
- оставить кнопку заблокированной;
- через 2–3 секунды перенаправить пользователя на страницу входа.

8. Не менять рабочую серверную логику токенов без необходимости.

==================================================
ЗАДАЧА 2. При регистрации эксперта не сохраняются связанные данные
==================================================

Сейчас при регистрации эксперт заполняет:

- регион;
- номер сертификата;
- направление или направления экспертизы;
- возможно срок действия сертификата и другие данные сертификата.

После регистрации:

- auth-пользователь создаётся;
- вход работает;
- профиль эксперта создаётся;
- личный кабинет открывается.

Но в базе не создаются записи в таблицах:

- palata_expert_regions;
- palata_expert_certificates;
- palata_expert_directions.

Из-за этого после первого входа в личном кабинете отсутствуют:

- регион;
- номер сертификата;
- срок действия сертификата;
- направления экспертизы.

При этом повторное внесение данных через «Редактировать профиль» может сохранять их, потому что редактирование использует другой API.

Нужно провести полный аудит именно регистрации эксперта.

Проверить цепочку:

форма регистрации
→ данные формы
→ API регистрации
→ создание auth-пользователя
→ создание профиля эксперта
→ запись региона
→ запись сертификата
→ запись направлений экспертизы.

Что необходимо сделать:

1. Найти файлы формы регистрации эксперта.

2. Найти API, который вызывается при регистрации эксперта.

3. Проверить, какие поля реально отправляет форма:

- region_id или массив region_ids;
- certificate_number;
- cert_valid_to;
- cert_expert_name;
- cert_direction_ids;
- direction_id или direction_ids;
- остальные поля сертификата, если они присутствуют.

4. Проверить, какие поля принимает backend.

5. Найти место, где создаётся запись в palata_expert_profiles.

6. Определить точную причину, почему связанные данные не записываются в:

- palata_expert_regions;
- palata_expert_certificates;
- palata_expert_directions.

7. После создания профиля эксперта сохранять связанные данные в реальные таблицы с реальными названиями колонок из текущей базы.

Для palata_expert_regions:

- expert_id;
- region_id.

Для palata_expert_certificates:

- expert_id;
- certificate_number;
- status;
- cert_valid_to;
- cert_expert_name;
- cert_direction_ids;
- остальные существующие поля, если они приходят из формы.

Для palata_expert_directions:

- expert_id;
- direction_id.

8. Обязательно проверить, какой именно идентификатор должен использоваться в expert_id.

Не перепутать:

- auth user id;
- user id;
- profile id;
- expert profile id;
- expert id.

Использовать тот идентификатор, который уже используется личным кабинетом и таблицами связей.

9. Создание профиля, региона, сертификата и направлений выполнить атомарно.

Если используется Prisma — применить одну транзакцию.

Если один из обязательных шагов не выполнен, регистрация не должна оставлять частично созданный профиль без связанных данных.

10. Не создавать дубли.

Для массивов регионов и направлений использовать подходящий вариант:

- createMany + skipDuplicates;
- upsert;
- deleteMany + createMany, если это соответствует текущей архитектуре.

11. Не ломать существующий вход, восстановление пароля и создание auth-пользователя.

12. Добавить диагностические логи без персональных данных и секретов:

[EXPERT-REGISTER] user created
[EXPERT-REGISTER] profile created
[EXPERT-REGISTER] regions saved
[EXPERT-REGISTER] certificate saved
[EXPERT-REGISTER] directions saved

Логировать:

- идентификаторы;
- количество записей;
- факт выполнения шага.

Не логировать:

- пароль;
- токены;
- полный номер сертификата;
- персональные данные.

ОБЯЗАТЕЛЬНЫЕ ТЕСТЫ
==================================================

Тест 1. Новая регистрация эксперта

После регистрации должны появиться:

- 1 auth-пользователь;
- 1 запись в palata_expert_profiles;
- минимум 1 запись в palata_expert_regions, если регион был выбран;
- минимум 1 запись в palata_expert_certificates, если сертификат был заполнен;
- минимум 1 запись в palata_expert_directions, если направление было выбрано.

Тест 2. Проверка данных

Проверить, что связанные записи содержат правильный expert_id.

Тест 3. Повторная регистрация с ошибкой

Если не удалось сохранить обязательные связанные данные, не должен оставаться частично созданный профиль.

Тест 4. Восстановление пароля

При недействительном токене пользователь видит понятный текст, а не HTTP 400.

Тест 5. Успешная смена пароля

После успешной смены пароля пользователь видит сообщение и автоматически переходит на страницу входа.
