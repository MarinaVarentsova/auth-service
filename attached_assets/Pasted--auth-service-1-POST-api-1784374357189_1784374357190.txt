КРИТИЧЕСКИ ВАЖНО:

UniSender sendEmail может вернуть минимум два формата успешного ответа.

Новый формат при error_checking=1:

{
  "result": [
    {
      "email": "user@example.com",
      "id": "message-id"
    }
  ]
}

Старый формат совместимости:

{
  "result": {
    "email_id": 14362456134
  }
}

Текущий lib/unisender.ts ожидает только data.result?.[0].id.
Из-за этого успешный ответ старого формата может ошибочно считаться ошибкой
"UniSender did not return email id", после чего register возвращает 502.

Нужно:

1. Найти точную фактическую причину 502.

2. В lib/unisender.ts логировать полный безопасный ответ UniSender:
   - HTTP status;
   - response body;
   - error/code;
   - формат result;
   - не выводить UNISENDER_API_KEY;
   - не выводить полный verification token;
   - email можно маскировать.

3. Исправить типы ответа UniSender так, чтобы поддерживались оба успешных формата:

   а) result как массив:
      data.result[0].id

   б) result как объект:
      data.result.email_id

4. Учесть, что email_id может быть number или string.

5. Если UniSender вернул:
   - HTTP error;
   - data.error;
   - errors у конкретного получателя;
   - пустой или неизвестный результат,
   выдать в серверный лог точную причину.

6. Проверить наличие переменных:

   UNISENDER_API_KEY
   UNISENDER_FROM_EMAIL
   UNISENDER_FROM_NAME
   UNISENDER_LIST_ID
   APP_URL

7. Не подставлять молча неправильные значения для критических переменных.
   Для API key и list id должна быть явная ошибка конфигурации.

8. Проверить, что verificationUrl формируется через APP_URL и содержит правильный путь существующего verify endpoint.

9. Проверить, что sendVerificationEmail действительно вызывается после создания verification token.

10. Исправить проблему частично созданного аккаунта.

Сейчас пользователь и verification token сохраняются, а затем письмо падает, после чего остаётся аккаунт, который мешает повторной регистрации.

Сделай безопасное компенсирующее удаление:

- если новый пользователь и verification token были созданы в текущем запросе;
- и sendVerificationEmail завершился ошибкой;
- удалить созданный verification token;
- удалить только созданного в этом запросе пользователя;
- не удалять ранее существующих пользователей;
- затем вернуть 502.

Удаление должно выполняться аккуратно и логироваться.
Если очистка тоже завершилась ошибкой, залогировать отдельно EMAIL_SEND_FAILED и REGISTRATION_ROLLBACK_FAILED.

11. Не держать транзакцию PostgreSQL открытой во время внешнего HTTP-запроса к UniSender.

12. Ответы API оставить понятными:

Успех:
201 и существующий успешный формат ответа регистрации.

Ошибка отправки:

  "message": "Не удалось отправить письмо подтверждения. Попробуйте зарегистрироваться ещё раз."
}

Конкретную внутреннюю ошибку UniSender не возвращать в браузер, только в серверные логи.
