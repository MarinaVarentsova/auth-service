Работай в основном репозитории платформы и отдельно в auth-service только там, где это необходимо для verify redirect.

Нужно исправить два бага без изменения бизнес-логики регистрации и профиля.

БАГ 1. После подтверждения email показывается JSON

Сейчас после клика по ссылке подтверждения открывается:

/api/auth/verify?token=...

и пользователь видит JSON:

{
  "success": true,
  "message": "Email confirmed"
}

Нужно после успешного подтверждения перенаправлять пользователя в интерфейс личного кабинета.

Проверь фактический файл:

auth-service:
.migration-backup/app/api/auth/verify/route.ts

или рабочий эквивалент этого route в production.

Требуемое поведение:

1. Проверить токен как сейчас.
2. Обновить пользователя как сейчас:
   - emailVerified = true
   - status = active
   - verification token = used
3. После успешного подтверждения делать redirect на APP_URL.

Использовать существующую переменную:

APP_URL

Не использовать:
- 0.0.0.0
- localhost
- AUTH_PUBLIC_URL для редиректа в интерфейс

Целевой URL после подтверждения:

${APP_URL}/login?emailVerified=1

Если в приложении уже есть более подходящий route после подтверждения, использовать его и указать в отчёте.

Важно:
после подтверждения пользователь ещё не авторизован, поэтому прямой redirect в защищённый /dashboard может вернуть его на login.
Поэтому безопасный вариант:
${APP_URL}/login?emailVerified=1

На странице login показать уведомление:

"Email подтверждён. Теперь вы можете войти."

Если пользователь уже имеет активную сессию и приложение умеет это определить, допускается redirect в его ЛК, но не ломать текущую auth-логику.

Для ошибок verify:
- недействительный токен;
- просроченный токен;
- уже использованный токен

не показывать сырой JSON в браузере.
Перенаправлять на:

${APP_URL}/login?emailVerified=0&reason=invalid_token

и отображать понятное сообщение.

БАГ 2. После первой регистрации эксперта в ЛК не отображаются регионы, направления экспертизы и сертификат

Фактическое поведение:

1. Эксперт заполняет форму регистрации.
2. Без региона и номера действующего сертификата регистрация эксперта невозможна.
3. После подтверждения email и первого входа профиль существует.
4. Но в ЛК отображается:
   - регионы: "Не указаны";
   - направления экспертизы: отсутствуют;
   - сертификат/образование: "Не заполнено".
5. Если открыть редактирование профиля, повторно выбрать регион, повторно указать сертификат и сохранить — всё появляется.

Это значит, что данные формы регистрации либо:
- не сохраняются в профильные таблицы;
- сохраняются не под тем user_id;
- сохраняются до появления palata_users;
- сохраняются, но GET профиля читает другие таблицы/поля;
- frontend первого входа читает устаревший или неполный endpoint.

Нужно провести полный аудит сценария регистрации эксперта:

Форма регистрации
→ auth-service создаёт auth_users
→ основной backend создаёт palata_users
→ создаёт palata_expert_profiles
→ создаёт palata_expert_regions
→ создаёт сертификаты/связи сертификата
→ определяет направления экспертизы
→ GET профиля эксперта
→ интерфейс ЛК

Проверь все связанные сущности и реальные таблицы:

- palata_users
- palata_expert_profiles
- palata_expert_regions
- palata_expert_certificates
- palata_certificates
- palata_expertise_directions
- palata_expert_directions
- таблицы специализаций, если используются
- регистрационный endpoint
- endpoint сохранения профиля
- endpoint получения профиля
- frontend-компонент ЛК эксперта

Нужно сравнить два сценария:

A. первичная регистрация эксперта;
B. редактирование профиля после входа.

Найди, какие записи создаёт сценарий B, но не создаёт сценарий A.

КРИТИЧЕСКОЕ ТРЕБОВАНИЕ:

Не дублировать бизнес-логику вручную в двух местах.

Если уже существует рабочий сервис сохранения профиля, который используется при редактировании, вынести или переиспользовать его для первичной регистрации.

При первичной регистрации должны быть сохранены:

1. palata_users:
   - auth user id или корректная связь с auth_users;
   - role = expert;
   - email;
   - full_name;
   - is_active.

2. palata_expert_profiles:
   - user_id;
   - experience_description;
   - принимает заказы;
   - готовность к командировкам;
   - участие в СРО;
   - номер регистрации СРО, если заполнен;
   - другие поля формы.

3. palata_expert_regions:
   - все выбранные регионы;
   - корректный expert_id/profile_id/user_id согласно текущей схеме.

4. Сертификаты:
   - номер сертификата;
   - тип/реестр;
   - срок действия, если есть;
   - связь с экспертом;
   - статус действующий.

5. Направления экспертизы:
   - должны определяться из действующего сертификата по существующей бизнес-логике;
   - не принимать произвольное направление только из фронтенда;
   - использовать существующий справочник и существующий механизм сопоставления сертификата.

Проверь порядок операций.

Возможная причина:
регистрационная форма отправляет профильные данные до того, как создана запись palata_users, либо использует auth user id вместо palata_users.id.

Нужно логировать безопасно:

[EXPERT-REGISTER] auth user resolved
[EXPERT-REGISTER] palata user created
[EXPERT-REGISTER] expert profile created
[EXPERT-REGISTER] regions saved
[EXPERT-REGISTER] certificate saved
[EXPERT-REGISTER] directions resolved
[EXPERT-REGISTER] registration completed

При ошибке показать:
- какой шаг упал;
- какие id использовались;
- не логировать пароль и секреты.

Добавить транзакцию для профильных данных в основной PostgreSQL:
- palata_users;
- expert profile;
- regions;
- certificate links;
- directions.

Auth-service и основная база — разные базы, поэтому не пытаться делать одну общую DB-транзакцию между ними.
Но внутри основной базы создание профиля эксперта должно быть атомарным.

Проверить GET профиля после создания:
сразу после регистрации endpoint должен возвращать те же данные, которые отображаются после ручного редактирования.

Добавить временный диагностический endpoint или тест только при необходимости.
Не оставлять его публичным после проверки.

Тест-кейс:

1. Удалить тестового пользователя из auth и Palata.
2. Зарегистрировать нового эксперта с:
   - регионом "Вся Россия";
   - действующим сертификатом;
   - направлением, определяемым сертификатом;
   - описанием опыта.
3. Подтвердить email.
4. Войти первый раз.
5. Без открытия редактирования профиля в ЛК должны отображаться:
   - регион;
   - направление экспертизы;
   - номер сертификата;
   - описание опыта;
   - настройки профиля.

Также проверить customer-регистрацию, чтобы исправление эксперта её не сломало.

После изменений выполнить:
- build основного приложения;
- build auth-service;
- typecheck;
- prisma validate для обеих схем, где используется Prisma.

В финальном отчёте показать:

1. Точную причину бага redirect после verify.
2. Точную причину отсутствия регионов/сертификата/направлений после первого входа.
3. Какие записи отсутствовали после регистрации.
4. Чем отличались сценарии регистрации и редактирования.
5. Какие файлы изменены.
6. Какие таблицы теперь заполняются.
7. Какой URL используется после подтверждения email.
8. Результаты build/typecheck/prisma validate.
9. Commit SHA для каждого репозитория.
